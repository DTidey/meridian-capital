# Data Pipeline Changes — 2026-06-15

## 1. Remove `fcf_yield` from fundamentals schema

**Problem:** `fcf_yield` in the `fundamentals` table was always `NULL`. It was hardcoded to `None` in Layer 1 with a comment saying Layer 2 would populate it, but that write-back was never implemented. The actual FCF yield calculation already existed in the factors table as `val_fcf_yield`.

**Fix:** Removed `fcf_yield` entirely from the fundamentals schema, computation, and all test fixtures. The canonical value remains `val_fcf_yield` in the factors table.

**Files changed:**
- `data/db.py` — column removed from `fundamentals` table definition
- `data/fundamentals.py` — removed stub assignment and dict key
- `tests/test_fundamentals.py` — removed null-check exception and `test_fcf_yield_is_always_none` test
- `tests/test_loader.py`, `tests/test_quality.py`, `tests/test_growth.py`, `tests/test_value.py` — removed field from fixture dicts

---

## 2. Fix `eps_estimate` always null in `earnings_calendar`

**Problem:** `_fetch_earnings_date` looked up `"Earnings Estimate"` from the yfinance calendar dict, but yfinance uses the key `"Earnings Average"` for the consensus EPS estimate. The fallback was always `[None]`, so `eps_estimate` was never populated. A secondary bug (`float(... or 0) or None`) would also have silently discarded a genuine `0.0` estimate.

**Fix:** Corrected the key to `"Earnings Average"` and removed the `or 0` coercion.

**Files changed:**
- `data/earnings_calendar.py` — key name corrected, coercion removed

---

## 3. Increase earnings calendar lookahead from 30 to 90 days

**Problem:** With a 30-day window, a large fraction of S&P 500 tickers (those reporting beyond 30 days) had no row in the `earnings_calendar` table at any given time, resulting in only ~217 rows instead of near-full universe coverage.

**Fix:** Increased `lookahead_days` from 30 to 90 in config.

**Files changed:**
- `config.yaml` — `earnings_calendar.lookahead_days: 30 → 90`

---

## 4. Fix earnings transcript ingestion (always fetched zero transcripts)

**Problem:** In `run_data.py`, the transcript step used `args.tickers or []` as the ticker list. On a normal full run `args.tickers` is `None`, so `update_transcripts` was always called with an empty list. The design intent was "candidates only", but candidates aren't known until after Layer 2 scoring — which runs after Layer 1 — so the wiring was impossible.

**Fix:** Removed the transcript step from `run_data.py` entirely and created a new `run_transcripts.py` script that runs between Layer 2 and Layer 3. It queries the `factor_scores` table for the current day's `LONG`/`SHORT` candidates and fetches transcripts only for those tickers, minimising FMP API calls.

**Files changed:**
- `run_data.py` — transcript step and import removed
- `run_transcripts.py` — new script created
- `run_all.py` — new "Transcript ingestion" step inserted between Layer 2 and Layer 3

---

## 5. Fix `content_text` quality in `sec_filings`

**Problem:** The HTML stripping in `_fetch_filing_text` used a simple tag-removal regex on the raw response. This left JavaScript, CSS, and inline iXBRL hidden-section content mixed into the stored text, producing very noisy input for the AI analyser.

**Fix:** Before stripping tags, now explicitly removes entire `<script>`, `<style>`, and `<ix:hidden>` blocks (content and all). Also added `html.unescape()` to resolve entities such as `&nbsp;` and `&amp;`.

**Files changed:**
- `data/sec_data.py` — `_fetch_filing_text` updated

---

## 6. Fix `fetched_at` always null in `sec_filings`

**Problem:** Rows written in an earlier version of the code (before `fetched_at` was being set) remained `NULL` indefinitely because `insert_or_ignore` never updates existing rows.

**Fix:** Added a one-time backfill at the start of `update_sec_data` that sets `fetched_at = filed_date` for any row where `fetched_at IS NULL`. New inserts were already setting the field correctly.

**Files changed:**
- `data/sec_data.py` — backfill `UPDATE` added to `update_sec_data`

---

## 7. Fix institutional holdings — no new data, and broken summary metrics

Three related bugs all caused by the same root issue: only one quarter of 13-F data was ever loaded.

### 7a. Only one quarter ever fetched

**Problem:** `_get_latest_13f_accession` returned only the single most recent 13-F filing. Once loaded, subsequent runs always found `existing > 0` and skipped. No historical quarters were ever stored, making quarter-on-quarter comparison impossible.

**Fix:** Renamed to `_get_recent_13f_accessions`, now returns up to 6 recent filings. `update_institutional` iterates over all of them, skipping quarters already in the DB and storing any that are new.

### 7b. `net_share_change` always null

**Problem:** `_summarise_holdings` computed a prior-quarter date by subtracting exactly 3 months from the filing date. Because `report_date` is the filing date (not the period end), this shifted slightly each quarter and the exact-match lookup against the holdings table always failed.

**Fix:** Replaced the date arithmetic with a dynamic query: `MAX(report_date) WHERE report_date < current`. This finds whatever prior quarter is actually present in the DB regardless of the exact date.

### 7c. `funds_holding` always equal to `new_positions`

**Problem:** With no prior quarter data in the DB, `prior_funds` was always an empty set, so `new_positions = len(current_funds - {}) = funds_holding`.

**Fix:** Resolved automatically once prior-quarter data is available (7a) and the prior-quarter lookup works correctly (7b).

**Files changed:**
- `data/institutional.py` — `_get_latest_13f_accession` replaced by `_get_recent_13f_accessions`; `update_institutional` loops over all recent filings; `_summarise_holdings` uses dynamic prior-quarter lookup; `_prior_quarter_report_date` removed
